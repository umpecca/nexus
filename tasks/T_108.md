# T_108: Add an optional ngrok tunnel for the MCP server

## Status

Done

## Summary

Add an optional, off-by-default ngrok tunnel that exposes the enabled embedded MCP server through a public ngrok URL so remote AI clients can reach it. The user supplies an ngrok authtoken and may optionally supply a reserved/custom ngrok domain (both saved per OS profile with the other MCP settings). The tunnel forwards to the loopback MCP port; the ngrok agent connects outbound, so no inbound port is opened and the MCP server still binds `127.0.0.1`. When a custom domain is set it is used; when it cannot be bound the tunnel falls back to a random URL and reports the fallback. Bearer-token authentication and the write-confirmation dialog continue to apply. Enabling the tunnel while the authentication mode is none shows a prominent warning but is allowed.

## Implementation Notes

- Settings (`src/lib/settings.ts`):
  - Extend `McpServerSettings` with `ngrokEnabled: boolean` (default `false`), `ngrokAuthToken: string` (default `""`), and `ngrokDomain: string` (default `""`).
  - Update `createDefaultMcpServerSettings` and `sanitizeMcpServerSettings`: coerce `ngrokEnabled` to a boolean (default `false`) and `ngrokAuthToken` / `ngrokDomain` to trimmed strings.
  - The authtoken is persisted in local settings (plaintext) by explicit user choice.
  - Extend `src/lib/settings.test.ts`: the default `mcpServer` now includes `ngrokEnabled: false` and `ngrokAuthToken: ""`; add a load-valid case and a sanitize-invalid case for the two fields; update the existing full-object `createDefaultSettings`, font-only load, and save assertions to include them.
- New module `electron/ngrok-tunnel.cjs` (isolates the `@ngrok/ngrok` SDK):
  - `ensureTunnel({ port, authToken, domain })`: start or restart a single forwarding tunnel to `localhost:<port>` via `ngrok.forward({ addr: port, authtoken: authToken, domain? })`, then read `listener.url()`. When `domain` is set, bind to it; if forwarding with the domain fails, retry once without the domain (fall back to a random URL) and set `domainFallback`. If the current tunnel is already connected with the same `port`, `authToken`, and `domain`, no-op and return the cached state. If any change, close the existing listener and re-forward. Guard against concurrent `ensureTunnel` calls (serialize with a pending promise).
  - `stopTunnel()`: close the active listener (if any) and clear state.
  - `getTunnelState()`: return `{ connected, url, error, domainFallback }`.
  - Catch start failures and store the message in `error`; never throw out of `ensureTunnel` in a way that crashes the configure flow.
- Main process (`electron/main.cjs`):
  - In the existing `mcp:configure` handler, accept `ngrokEnabled` and `ngrokAuthToken` from the config payload (sanitize: boolean and string).
  - After `mcpServer.configure(...)` reports the server is listening: if the server is enabled and listening, `ngrokEnabled` is true, and `ngrokAuthToken` is non-empty, call `ensureTunnel({ port, authToken })`; otherwise call `stopTunnel()`.
  - Extend the handler result with `ngrok: { enabled, connected, url, error }` from `getTunnelState` (plus the requested `enabled` flag) so the renderer can show the public URL or an error.
  - Stop the tunnel on `will-quit` alongside `mcpServer.stop()`.
  - A tunnel failure must not stop or block the local MCP server.
- Preload (`electron/preload.cjs`) and types (`src/electron.d.ts`):
  - `configureMcpServer` passes `ngrokEnabled` and `ngrokAuthToken` in `ConfigureMcpServerInput`.
  - Extend `ConfigureMcpServerResult` with an `ngrok` status object (`enabled`, `connected`, `url`, `error`).
- App shell (`src/App.tsx`):
  - Include `ngrokEnabled` and `ngrokAuthToken` in the existing `mcp:configure` effect payload and its dependency array.
  - Store the returned `ngrok` status in state and pass it to `SettingsDialog`. Keep the existing MCP configure behavior intact.
- Settings dialog (`src/components/settings/SettingsDialog.tsx`):
  - Inside the existing MCP server section, add an "Expose via ngrok tunnel" checkbox bound to `mcpServer.ngrokEnabled`, an ngrok authtoken input bound to `mcpServer.ngrokAuthToken`, and an optional custom-domain input bound to `mcpServer.ngrokDomain`, all routed through the existing `onMcpServerChange`.
  - Show a notice when a custom domain is set but the runtime ngrok status reports `domainFallback` (the requested domain was unavailable and a temporary URL is in use).
  - When the runtime ngrok status reports connected, show the public URL and the public MCP endpoint URL (`url` + `/mcp`), each with a copy-to-clipboard control. Show the tunnel error message when present.
  - Show a prominent warning when `ngrokEnabled` is on while `authMode` is `none`, explaining that an unauthenticated MCP server would be exposed to the public internet.
  - Reuse the existing MCP settings field, warning, and copy-button styles.
- Packaging (`package.json`):
  - Add an electron-builder `asarUnpack` entry so the `@ngrok/ngrok` native binary is available at runtime in packaged builds (for example `**/node_modules/@ngrok/ngrok/**`).
