# T_124: Encrypt the MCP bearer token at rest (safeStorage)

## Status

Done

## Summary

The MCP bearer token authenticates AI clients (and the OAuth flow issues it), making it the most
powerful secret the app holds, yet it was stored in plaintext in renderer localStorage. Give it the
same treatment the QuickConnect token already received in T_112: encrypt it at rest with Electron
`safeStorage` (DPAPI / Keychain / libsecret) in a per-profile file under userData, keep only an
in-memory copy in the renderer/main process, and migrate any existing plaintext token on first launch
after upgrading.

## Implementation Notes

- `src/lib/settings.ts`:
  - `saveSettings` now strips `mcpServer.bearerToken` (writes `""`) before serializing to localStorage,
    so the token is never persisted there. The in-memory settings object still carries it, so all
    consuming code (configure effect, settings dialog) is unchanged — only persistence differs. Key
    order is preserved, so the serialization-shape test is unaffected.
  - Add `readLegacyMcpBearerToken(profileName)`, mirroring `readLegacyQuickConnectToken`, to read a
    plaintext token an older version left in the settings JSON for one-time migration.
- `electron/main.cjs`: add `mcp:get-bearer-token` / `mcp:set-bearer-token` IPC handlers and a
  `mcp-bearer-tokens.json` per-profile ciphertext store, a direct parallel of the QuickConnect token
  handlers. An empty token clears the entry; when `safeStorage.isEncryptionAvailable()` is false the
  token is not written (never plaintext on disk), matching the QuickConnect security stance.
- `electron/preload.cjs` + `src/electron.d.ts`: expose `getMcpBearerToken` / `setMcpBearerToken`
  (reusing the `{ stored, encryptionAvailable }` result shape).
- `src/App.tsx`:
  - On profile init, capture any legacy plaintext token, then after `loadSettings` load the token from
    the encrypted store (migrating the legacy value once) and merge it into `settings.mcpServer`.
    Record it in `persistedBearerTokenRef` so it is not written straight back.
  - A persist effect writes the token to the encrypted store whenever it genuinely changes
    (enable/regenerate), guarded by the ref and skipping empty values so it never clears the store and
    never races the startup hydration.

## Notes

- When `safeStorage` is unavailable (e.g. a headless Linux without libsecret) the token is kept in
  memory for the session but not persisted, so it must be re-generated next launch — the deliberate
  "never plaintext on disk" trade-off shared with the QuickConnect token. On Windows/macOS safeStorage
  is available.
- Regenerating the token still revokes every client (including OAuth-issued access), unchanged.
- Reset-to-defaults clears the in-memory token but, like the QuickConnect token, does not wipe the
  encrypted store; the value re-hydrates next launch. Left consistent with the existing behavior.

## Verification

- `npm run build` (`tsc --noEmit` + vite) and `npm run test` pass (288 tests).
- `src/lib/settings.test.ts`: added cases asserting `saveSettings` never writes the bearer token to
  localStorage (persists `""` while keeping the other MCP fields), and that `readLegacyMcpBearerToken`
  reads a legacy plaintext token and returns `""` for missing/non-string values.
- `node --check` on the edited `.cjs` files. The encrypt/decrypt path itself reuses the proven
  QuickConnect handlers (safeStorage is unavailable under vitest, so it is covered manually).
- Manual (desktop): enable the MCP server, copy the token, quit and relaunch → the same token loads
  and the server starts; confirm `mcp-bearer-tokens.json` under userData contains ciphertext (not the
  token) and the settings localStorage entry's `mcpServer.bearerToken` is empty; an existing plaintext
  token from before the upgrade migrates on first launch.
