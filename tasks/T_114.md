# T_114: Add a Recent Files / Open Recent menu

## Status

Done

## Summary

Every Nexus session previously started from scratch with no memory of what the user had been editing —
reopening a document meant navigating the filesystem again. This adds a persisted recent-files list
surfaced as a File/Open Recent submenu. Opening or saving a document records its path; the list is
most-recent-first, deduped (case-insensitively on Windows), capped at ten, persisted by the main
process across sessions, and mirrored to the operating system's recent documents. Selecting an entry
reopens it with the same Save/Don't Save/Cancel confirmation used by File/Open, a Clear Recent command
empties the list, and an entry whose file has gone missing is pruned with a native error dialog.

## Implementation Notes

- New `electron/recentFiles.cjs`: pure, Electron-free helper holding the list logic so it is unit
  testable. Exports `DEFAULT_RECENT_FILES_LIMIT` (10), `defaultComparePath` (resolves and lowercases
  only on `win32`), `sanitizeRecentFiles`, `addRecentFile`, and `removeRecentFile`. Each list function
  takes an `{ limit, comparePath }` options object, returns a new deduped/capped most-recent-first
  array, and never mutates its input.
- New `electron/recentFiles.test.ts`: 16 vitest cases covering front-promotion, dedupe, capping,
  no-mutation, empty/invalid paths, sanitization of corrupt/non-array input, and platform-specific
  comparison. Lives under `electron/` (outside the `tsc` `src` include) so it does not affect the
  renderer typecheck; vitest's default glob still discovers it.
- `electron/main.cjs`:
  - `require("./recentFiles.cjs")`; keep an in-memory `recentFiles` array plus a `recentFilesOptions`
    `{ limit, comparePath: getComparableFilePath }` so dedupe matches the app's existing path
    comparison. `loadRecentFiles` (sync `readFileSync` at launch) sanitizes the persisted JSON;
    `persistRecentFiles` writes `recent-files.json` under `app.getPath("userData")` (best-effort).
  - `addRecentFile` delegates list computation to the helper, and on an actual change updates the
    array, calls `app.addRecentDocument`, persists, and rebuilds the menu (skipping the rebuild when
    the list is unchanged so repeated saves of the same file do not churn the menu). `removeRecentFile`
    and `clearRecentFiles` (the latter also calling `app.clearRecentDocuments`) follow the same shape.
  - Record on every successful `readMarkdownFile` (covers File/Open, watched re-read, recent open, and
    OS handoff) and after a successful `file:saveAs`.
  - Add `buildRecentFilesSubmenu` (file-name items with the full path as tooltip, a disabled
    "No Recent Files" placeholder when empty, a separator, and "Clear Recent") and an `Open Recent`
    item in the File submenu after `Open`. `sendOpenRecentFile` posts the path to the focused renderer
    over a `menu:open-recent` channel, falling back to opening a new window when no window is focused.
  - New `recent:open` IPC handler: reads the path (recording it), refreshes the watch signature, and on
    failure prunes the entry and shows a native error dialog (mirroring `file:get-initial-open-file`).
  - Call `loadRecentFiles()` immediately before the first `buildMenu()` in `app.whenReady`.
- `electron/preload.cjs`: add an `onOpenRecentFile(callback)` subscription on `menu:open-recent` and an
  `openRecentFile(filePath)` invoker for `recent:open`.
- `src/electron.d.ts`: add `onOpenRecentFile` and `openRecentFile` to the `window.nexus` surface.
- `src/App.tsx`: add an `openRecentFile(filePath)` handler that runs `confirmDirtyBufferAction` then
  loads via `window.nexus.openRecentFile`; register it in both `menuHandlersRef` objects and subscribe
  to `onOpenRecentFile` in the menu-listener effect (with cleanup).
- Docs: update `PRODUCT.md` (capability, requirements, primary workflow) and `ARCHITECTURE.md`
  (main/preload/`recentFiles.cjs` component descriptions, flow items 64–65, a Recent Files Workflow
  subsection, and a recent-files data store), plus this changelog entry.

## Verification

- `npm run build` (`tsc --noEmit` + vite) and `npm run test` (now 81 tests, including the 16 new
  `recentFiles` cases) pass. `node --check` passes for `electron/main.cjs`, `electron/preload.cjs`, and
  `electron/recentFiles.cjs`.
- Boot smoke test: launched the packaged main process; it survives past `app.whenReady()` with no
  errors, confirming the new module load, `loadRecentFiles()`, the Open Recent submenu build, and the
  `recent:open` registration all execute cleanly.
- Manual test (desktop): open several files, confirm they appear most-recent-first under
  File/Open Recent; reopen one and confirm the dirty-buffer prompt appears when the buffer is unsaved.
- Manual test: Save As a new document and confirm it joins the list; reopen the same file and confirm
  it moves to the top without duplicating.
- Manual test: restart the app and confirm the list persists; use Clear Recent and confirm the submenu
  empties; delete a listed file on disk, choose it, and confirm the error dialog plus its removal.
