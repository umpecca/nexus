# T_097: Add the MCP replace_document tool with diff confirmation modal

## Status

Done

## Summary

Activate the write side of the embedded MCP server. Replace the stub `nexus_replace_document` tool from T_096 with a real implementation that routes through the focused renderer's new shadcn-styled MCP write confirmation dialog. The user must approve every individual call before the editor buffer is replaced.

## Implementation Notes

- Replace the stub tool registration with a real handler in `electron/mcp-server.cjs`. Args: optional `windowId` string and required `markdown` string. Behavior:
  - Resolve the target window (focused if `windowId` is omitted). If no window is available, return a JSON-RPC error result with `applied: false` and `reason: "no-window"`.
  - Create a pending pending-write record keyed by a fresh request ID, send `mcp:confirm-write` IPC to the target renderer with `{ requestId, markdown, clientLabel }` (clientLabel can be derived from `initialize` client info — store it in the server module).
  - Wait for the renderer to reply via `mcp:write-decision` `{ requestId, decision: "approve" | "reject", reason? }`. Use a Promise stored in a map keyed by `requestId`.
  - On approve, return `{ applied: true }` to the MCP client. On reject, return `{ applied: false, reason: "user-rejected" }`.
- Reject pending writes automatically when:
  - The target renderer unregisters (window closed) — resolve as `{ applied: false, reason: "window-closed" }`.
  - The MCP server is disabled via `mcp:configure` — resolve all pending writes as `{ applied: false, reason: "server-disabled" }` and clear the map.
  - A second write for the same window arrives while a confirmation is already open — return `{ applied: false, reason: "busy" }` immediately without queuing.
- Add `src/components/mcp/McpWriteConfirmDialog.tsx`: shadcn-styled `Dialog` with a header ("Allow MCP write?"), a one-line description naming the client, an MDXEditor-driven diff between current buffer and proposed Markdown (use `diffSourcePlugin` initialized with the current buffer as baseline and the proposed Markdown loaded into the editor, read-only), and two buttons: Reject (default) and Approve.
- Add `App.tsx` state: `pendingMcpWrite: { requestId: string; markdown: string; clientLabel: string } | null`. Subscribe to a new `nexus.onMcpConfirmWrite(callback)` preload event. When invoked, set the state to open the dialog. On the dialog's Approve, call `loadDocument(proposedMarkdown, /* keep filePath */)` using the same programmatic-change guard already used for external reloads so stale `onChange` events do not mark the new content dirty before comparison settles, then call `nexus.resolveMcpWrite(requestId, "approve")`. On Reject or dialog close, call `nexus.resolveMcpWrite(requestId, "reject")`.
- Make sure the dialog cannot be dismissed by pressing Escape without triggering Reject so a request can never silently hang.
- Add the missing preload methods: `onMcpConfirmWrite`, `resolveMcpWrite`, plus the registration/state APIs from T_096 if any were deferred.
- Update the architecture data flow already documented in ARCHITECTURE.md so the implementation matches steps 49–51.
- Extend `mcp:configure` to call back into the renderer (via `mcp:server-state`) when the server starts, stops, or reports a bind error so the settings dialog can later display a status badge (small improvement: render "Listening on …" or "Port 39125 is already in use" beneath the toggle).
