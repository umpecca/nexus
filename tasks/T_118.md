# T_118: Add MCP in-buffer write tools (apply_edits, replace_section, set_frontmatter)

## Status

Done

## Summary

Widen the embedded MCP server's write surface beyond the full-buffer `nexus_replace_document` with
three granular, in-buffer write tools so an AI client can make targeted edits that pair with the
read tools from T_117 instead of resending the whole document:

- `nexus_apply_edits` — ordered string-anchored find/replace edits (`{ find, replace, all?, isRegex? }`).
  An edit that matches nothing, or matches more than once without `all`, fails the whole batch without
  changing the document — so a stale read is rejected rather than mis-applied. Empty `replace` deletes.
- `nexus_replace_section` — replace a whole section (heading through the line before the next
  same-or-higher heading) selected by `index`, `slug`, or `heading`; empty `markdown` deletes it.
- `nexus_set_frontmatter` — set/merge/remove scalar YAML frontmatter fields, creating the `---` block
  when absent and dropping it when the last field is removed.

The enabling fact is that `McpWriteConfirmDialog` is content-agnostic (it diffs the current buffer
against a proposed buffer). So each tool reduces to "compute a proposed full buffer from the cached
document, then route it through the existing confirmation/apply pipeline." No new trust surface, and
filesystem tools (save/open/export) stay out of scope.

## Implementation Notes

- `electron/mcpDocumentTools.cjs`: extract the section-range logic into an exported
  `getSectionRange(markdown, selector)` (returns `{ found, target, lines, startLine0, endLine0,
  headings }`); refactor `getDocumentSection` to use it; also export `parseHeadings` and
  `toPublicHeading` so the edits module reuses them.
- `electron/mcpDocumentEdits.cjs` (new, pure): `applyEdits`, `replaceSection`, `setFrontmatter`, each
  returning `{ ok: true, markdown }` or `{ ok: false, reason, ... }` (`anchor-not-found`, `ambiguous`,
  `invalid-regex`, `section-not-found`, `no-headings`, `frontmatter-unsupported`, `unsupported-value`,
  …). Literal matching counts via `split`; regex matching guards zero-width matches; frontmatter
  parsing only accepts simple `key: value` scalars and bails (leaving the block untouched) otherwise.
- `electron/main.cjs`: extract `requestReplaceWithMarkdown(record, proposedMarkdown, clientLabel)`
  from the old `requestReplaceDocument` closure (the `mcpPendingWrites` + `mcp:confirm-write` +
  busy/no-window handling), and add `requestComputedWrite(windowId, clientLabel, computeProposed)`
  that resolves the window, computes the proposed buffer, returns the edit error on failure (no dialog)
  or routes the proposal through the shared confirm path. Register `requestApplyEdits`,
  `requestReplaceSection`, `requestSetFrontmatter` (and the now-slimmer `requestReplaceDocument`) on
  the host.
- `electron/mcp-server.cjs`: add a `WRITE_TOOLS` array (the three tool definitions) included in
  `tools/list`, dispatch branches that validate input, and helpers `describeEditError` /
  `writeResultToToolContent` that map a host result to tool content — applied/rejected as data, and
  no-window/busy/send-failed/edit-failed as `isError`.
