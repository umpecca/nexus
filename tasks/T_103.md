# T_103: Remove the File / Export as Word Document feature

## Status

Done

## Summary

Tear out everything added in T_099 (initial DOCX export), T_100 (image / Mermaid / formatting fixes), T_101 (TurboDocx library swap), and T_102 (Marked renderer crash fix). After T_102 the export no longer crashes, but the produced .docx is functionally blank in Word/LibreOffice (file opens, pages exist, no body content), and rather than continue debugging the post-T_102 Electron pipeline we are removing the feature for now. HTML and PDF export must continue to work unchanged, including Mermaid SVG rendering for HTML/PDF.

## Implementation Notes

- **`package.json`**: Remove `@turbodocx/html-to-docx`. Run `npm install` to refresh `package-lock.json`.
- **`electron/main.cjs`**:
  - Remove `const htmlToDocx = require("@turbodocx/html-to-docx");`.
  - Remove the `docxInlineStyles` frozen object and `getAdmonitionInlineStyle` helper.
  - Revert `renderMarkdownAdmonitions(markdown, parseMarkdown)` so it no longer accepts an `options.inlineStyles` argument; drop the inline `style="..."` emission on `<aside>` and the title `<div>`.
  - Revert `createMarkedHighlightExtension()` so it no longer takes options and always emits `<mark>${inner}</mark>` with no inline style.
  - Revert `renderMarkdownExportHtml(markdown, currentPath, options)` so it no longer threads `inlineStyles`. Keep `renderer.image` and the Mermaid branch of `renderer.code` (those existed before T_099 and are required by HTML/PDF). Remove the `renderer.blockquote`, `renderer.table`, and `renderer.tablecell` overrides that T_100 added. Simplify the `renderer.code` override so the non-Mermaid branch returns `false` (Marked falls back to its default), and remove the `defaultCodeRenderer` / `defaultBlockquoteRenderer` / `defaultTableRenderer` / `defaultTablecellRenderer` captures.
  - Remove the `inlineLocalImageDataUrls(webContents)` and `rasterizeExportSvgsViaCapture(exportWindow)` helpers.
  - Revert `createExportWindow()` to its no-argument shape (drop the `sizeOptions` parameter). The PDF path already used the default behavior.
  - Remove `docxFilters`, `getDocxPageOrientationFromValue`, and `getDocxPageSetup`.
  - Remove the entire `ipcMain.handle("file:export-docx", …)` handler, including the diagnostic logging added in the T_102 follow-up.
  - Remove the `Export as Word Document…` menu item (`label: "Export as Word Document"`, `click: () => sendMenuAction("exportDocx")`) from `buildMenu`. Keep `Export as HTML…` and `Export as PDF…`.
- **`electron/preload.cjs`**: Remove `exportMarkdownAsDocx(currentPath, markdown, options)` bridge that wraps `file:export-docx`.
- **`src/electron.d.ts`**: Remove `"exportDocx"` from `NexusMenuAction`, the `ExportMarkdownDocxOptions` type alias, and `exportMarkdownAsDocx(...)` from the `window.nexus` declaration.
- **`src/App.tsx`**: Remove the `exportDocumentAsDocx()` function, the `exportDocx` field from the menu handlers ref, and the `case "exportDocx":` branch from the menu action subscriber switch.
- **`PRODUCT.md`**: Walk back the DOCX-related lines (in-scope bullets, out-of-scope bullets, the File menu item list under functional requirements, the export-specific functional requirements, the Edge Cases bullet about Word rasterization, and the Specialized Logic / Modes Export entry). Net effect: PRODUCT.md should describe HTML + PDF export only, with `.docx` named explicitly in the out-of-scope list.
- **`ARCHITECTURE.md`**: Walk back DOCX references in the `electron/main.cjs` Project Structure note, the data-flow steps that describe the DOCX path, the Export Workflow component description, the Export Workflow Technologies line, and the Technology Used `@turbodocx/html-to-docx` bullet. The Export Workflow component name reverts to "HTML and PDF Export".
- **`CHANGELOG.md`**: Add a `T_103: Remove the File / Export as Word Document feature.` entry under Added (matches the existing convention) and a one-line Changed entry noting the removal of `.docx` export, the `@turbodocx/html-to-docx` dependency, and the supporting renderer / serializer / rasterization helpers.
- Keep task files T_099, T_100, T_101, and T_102 as historical records of the work that was done and is now being reverted.
