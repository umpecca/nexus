# T_117: Add MCP read tools for outline, section, search, and selection

## Status

Done

## Summary

Widen the embedded MCP server's read surface from the original `nexus_list_windows` /
`nexus_get_document` pair with four navigation-oriented tools so an AI client can understand and
target a document without ingesting (or replacing) the whole buffer:

- `nexus_get_outline` — the heading tree (level, text, unique slug, ordinal index, 1-based line).
- `nexus_get_section` — the Markdown of one section, selected by heading `index`, `slug`, or
  `heading` text, running from the heading through the line before the next same-or-higher heading
  (deeper subsections included).
- `nexus_search_document` — literal or regex search returning 1-based line/column positions, a line
  preview, the accurate total, and a truncation flag (capped at `maxResults`, default 200, max 1000).
- `nexus_get_selection` — the text the user currently has highlighted, plus the editor mode.

The first three are served synchronously from the main-process MCP record's cached Markdown (no
renderer round trip), mirroring `nexus_get_document`. `nexus_get_selection` needs live renderer state,
so it routes through a request/response IPC pair like the write-confirmation flow. All four are
read-only; none touch the write-confirmation gate or change the document.

## Implementation Notes

- `electron/mcpDocumentTools.cjs` (new): pure, cache-served logic — `buildDocumentOutline`,
  `getDocumentSection`, `searchDocument`. The heading parser mirrors `src/lib/outline.ts`'s
  `extractOutline` (same ATX/fence regexes and `normalizeHeadingText`) but emits 1-based line numbers
  and a deduped slug per heading via the shared `electron/headingSlugger.cjs`. Section slicing runs to
  the next heading of the same or higher level and trims trailing blank lines; search compiles a
  literal-escaped or raw regex, guards zero-width matches against infinite loops, and counts the true
  total while capping the returned matches.
- `electron/mcp-server.cjs`: add the four tool definitions to `READ_ONLY_TOOLS` (so `tools/list`
  advertises them) and dispatch branches in `dispatchToolCall`. `nexus_get_section` requires exactly
  one selector and `nexus_search_document` requires a non-empty `query`; both wrap the host call in
  try/catch to surface validation/regex errors as `isError` tool content. Added a shared
  `noWindowMessage(windowId)` helper and reused it for `nexus_get_document`.
- `electron/main.cjs`: `require` the new module; add `resolveMcpWindowRecord` (shared by the read
  methods, also adopted by `mcpGetDocument`); add host methods `mcpGetOutline` / `mcpGetSection` /
  `mcpSearchDocument` that compose the cached record with the pure tools, plus `mcpRequestSelection`
  (a `mcp:request-selection` IPC send with a pending-resolver map, a 5s timeout, and `finishMcpSelection`
  single-settle). Wire all four onto the host passed to `mcpServer.setHost`. Add the
  `mcp:selection-result` IPC handler (sanitized via `normalizeSelectionResult`) and reject pending
  selections for a webContents in both window-teardown paths (`closed` and `mcp:unregister-window`).
- `electron/preload.cjs`: expose `onMcpRequestSelection` (subscribe to `mcp:request-selection`) and
  `resolveMcpSelection` (send `mcp:selection-result`).
- `src/electron.d.ts`: add `McpRequestSelectionEvent`, the exported `McpEditorSelection`, and the two
  new bridge methods.
- `src/App.tsx`: add `readEditorSelection()` (scoped to the editor surface via `editorSurfaceRef`,
  reporting `currentViewModeRef` mode and the DOM selection text only when the anchor is inside the
  editor) and an effect that answers `onMcpRequestSelection` with it.
