# T_119: Add an opt-in MCP auto-approve-writes setting

## Status

Done

## Summary

Every MCP write tool (full replace and the granular in-buffer edits from T_118) normally shows the
in-app diff confirmation dialog and applies only after the user clicks Approve. Add a per-OS-profile
setting that, when enabled, applies MCP writes immediately without the dialog — for trusted local
sessions and agent loops — while keeping confirmation as the default. The setting carries a prominent
warning because, combined with the ngrok tunnel, it would let a remote client change the document
without review.

## Implementation Notes

- `src/lib/settings.ts`: add `autoApproveWrites: boolean` to `McpServerSettings`; default `false` in
  `createDefaultMcpServerSettings`; sanitize in `sanitizeMcpServerSettings` (non-boolean → `false`).
- `src/lib/settings.test.ts`: add `autoApproveWrites: false` to the full-shape `toEqual` expectations
  and the `saveSettings` serialization expectation (placed after `bearerToken` to match the model's
  key order, since `saveSettings` stringifies the settings object directly), plus a load/sanitize test.
- `src/components/settings/SettingsDialog.tsx`: add a `handleMcpAutoApproveWritesChange` handler and,
  in the MCP section (shown when the server is enabled, above the ngrok controls), an "Auto-approve
  writes" checkbox with a `nexus-settings-warning` that escalates its wording when the ngrok tunnel is
  on.
- `src/App.tsx`: track the current flag in `autoApproveMcpWritesRef` (assigned in the render body like
  `filePathRef`, so the once-registered `onMcpConfirmWrite` listener reads the latest value). Extract
  `applyApprovedMcpWrite(requestId, markdown)` from `approvePendingMcpWrite` and, in the
  `onMcpConfirmWrite` handler, call it directly (skipping the `pendingMcpWrite` dialog state) when the
  ref is true. The write still flows through the existing programmatic-change guard so dirty state
  settles correctly.

The decision is made in the renderer, where the setting already lives, so no change to the
main-process MCP config sync was needed. The main process serializes writes per window (the busy
check), so a second write cannot arrive while one is pending.

## Verification

- `npm run build` (`tsc --noEmit` + vite) and `npm run test` pass (264 tests, incl. the new
  settings default/sanitize case and the updated shape expectations).
- Manual (desktop): with auto-approve off, an MCP write shows the diff dialog (unchanged). Enable
  "Auto-approve writes" in Settings → the next MCP write applies immediately with no dialog and the
  document visibly updates. Turn on the ngrok tunnel and confirm the warning escalates. Reload and
  confirm the flag persists per profile.
