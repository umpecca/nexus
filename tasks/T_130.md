# T_130: Add visual OpenAPI request-body editing

## Status

Done

## Summary

Add a Request Body section to each OpenAPI route operation so JSON and other body payloads can be
defined without switching to Markdown source mode.

## Acceptance Criteria

- A route operation can add or remove its OpenAPI 3 `requestBody` object.
- The visual form edits the request-body description and required flag.
- Users can add, rename, and remove request content/media types, defaulting to `application/json`.
- Each media type can reference an existing component schema or use an inline object schema.
- Inline object schemas support adding, renaming, removing, describing, typing, and marking properties
  as required.
- A JSON example editor validates input before updating the OpenAPI object and reports an actionable
  inline error for invalid JSON.
- Imported request bodies, media nodes, schemas, and examples retain unsupported fields and vendor
  extensions unless the user explicitly replaces or removes the owning value.
- The rich-text API reference immediately reflects saved request-body changes.
- Tests cover add/remove, media-type rename, component-schema selection, inline required properties,
  JSON example validation, and passthrough preservation.
- PRODUCT, ARCHITECTURE, CHANGELOG, and this task record are updated.

## Implementation Notes

- Keep request-body mutation logic in direct, testable helpers shared by the React form and tests.
- Continue using OpenAPI 3 `requestBody`; do not add the obsolete Swagger 2 `in: body` parameter form.

## Verification

- `npm run lint` passes.
- `npm run build` passes.
- Local visual smoke test: created a POST route, added the request body, added an inline field, and
  verified that invalid JSON example text shows an actionable error without changing the YAML draft.
