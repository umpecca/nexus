# T_100: Fix Word export so images, Mermaid diagrams, and text formatting survive the .docx conversion

## Status

Done

## Summary

The initial DOCX export from T_099 produced .docx files that were missing all images, all Mermaid diagrams, and most non-default text formatting. Root cause: `html-to-docx` cannot read `file://` image sources, cannot apply CSS rules from `<style>` blocks, and the previous canvas-based SVG rasterization tainted on Mermaid 11's `<foreignObject>` text labels. This task swaps the SVG rasterizer for `webContents.capturePage`, inlines local image bytes as base64 `data:` URLs, and threads an `inlineStyles` option through the export HTML pipeline so the DOCX path emits inline `style="..."` attributes while HTML and PDF stay unchanged.

## Implementation Notes

- Added `fileURLToPath` to the existing `node:url` import in `electron/main.cjs`.
- Added a frozen `docxInlineStyles` object plus `getAdmonitionInlineStyle(type)` helper that mirror the CSS rules from `buildExportHtmlDocument`'s `<style>` block.
- Threaded an `inlineStyles` option through `renderMarkdownAdmonitions`, `createMarkedHighlightExtension`, and `renderMarkdownExportHtml`. When set, the admonition renderer adds `style="..."` to the `<aside>` and title `<div>`; the highlight extension wraps `<mark>` with yellow-background inline styling; new Marked renderer overrides inject inline styles on `<pre>`, `<code>`, `<blockquote>`, `<table>`, `<th>`, and `<td>`. HTML and PDF callers do not pass the flag and their output is unchanged.
- Replaced `rasterizeExportSvgsAsImages` with `rasterizeExportSvgsViaCapture(exportWindow)` that walks every `<svg>`, scrolls it into view, resizes the export window if the diagram overflows, calls `webContents.capturePage(rect)` to get a `NativeImage`, encodes it as a `data:image/png;base64,...` URL, and replaces the SVG with an `<img>`.
- Added `inlineLocalImageDataUrls(webContents)` that finds every `<img src^="file:">`, reads the file via `fs.readFile` (resolved with `fileURLToPath`), MIME-detects with the existing `imageMimeTypes` map, base64-encodes, and rewrites the `src` attribute. HTTP(S) and existing `data:` URLs are not touched.
- `createExportWindow` now accepts optional `{ width, height }`; default behavior unchanged so PDF export output stays byte-identical. The DOCX handler passes `{ width: 1024, height: 768 }`.
- The `file:export-docx` IPC handler now passes `inlineStyles: true` to `renderMarkdownExportHtml`, calls `inlineLocalImageDataUrls` and `rasterizeExportSvgsViaCapture` after Mermaid renders, then serializes and feeds the HTML to `html-to-docx` as before.
- Per-element try/catch in both helpers so a single broken image or unrenderable SVG does not abort the whole export.
