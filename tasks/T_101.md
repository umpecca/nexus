# T_101: Switch the Word export pipeline to the actively-maintained TurboDocx fork of html-to-docx

## Status

Done

## Summary

The DOCX export added in T_099 and hardened in T_100 currently depends on the original `html-to-docx` (npm `html-to-docx@1.8.0`), which has not had a release in roughly four years. Swap the underlying HTML-to-DOCX converter to the TurboDocx-maintained fork (`@turbodocx/html-to-docx`, https://github.com/TurboDocx/html-to-docx, currently 1.21.0). Keep every DOCX-specific workaround the existing pipeline already relies on — Mermaid SVG-to-PNG rasterization via `webContents.capturePage`, local `file://` image inlining via base64 `data:` URLs, and inline `style="..."` attributes on the Marked-rendered HTML — because none of those limitations are specific to the unmaintained fork and the user has explicitly asked for Mermaid-as-image plus preserved text formatting.

## Implementation Notes

- In `package.json`, remove `"html-to-docx": "^1.8.0"` from `dependencies` and add `"@turbodocx/html-to-docx": "^1.21.0"`. Reinstall so `package-lock.json` regenerates.
- In `electron/main.cjs`, change line 9 from `const htmlToDocx = require("html-to-docx");` to `const htmlToDocx = require("@turbodocx/html-to-docx");`. Also update the comment near the `docxInlineStyles` block (currently around line 791) so the package reference reads `@turbodocx/html-to-docx`.
- The TurboDocx fork keeps the same call signature `htmlToDocx(htmlString, headerHTMLString, documentOptions, footerHTMLString)` and the same documentOptions surface (`orientation`, `pageSize`, `margins`, `table`, `footer`, `header`), so the existing `ipcMain.handle("file:export-docx", ...)` body needs no further structural changes. The fork still returns a `Buffer` (or `ArrayBuffer` in some build modes); the existing `Buffer.isBuffer(docxBuffer) ? docxBuffer : Buffer.from(await docxBuffer.arrayBuffer())` guard already covers both shapes.
- Mermaid handling stays unchanged: continue to rasterize every `<svg>` to PNG through `webContents.capturePage` for the DOCX path so the diagram renders in Microsoft Word and LibreOffice Writer without requiring inline-SVG support in the recipient.
- Text formatting (admonition callouts, code/pre, blockquote, `==highlight==` marks, tables and table cells) continues to flow through the inline `style="..."` attributes emitted by `renderMarkdownExportHtml({ inlineStyles: true, ... })`, because the converter still does not apply rules from `<style>` blocks. Keep `docxInlineStyles` and `getAdmonitionInlineStyle` exactly as T_100 added them.
- Update `ARCHITECTURE.md` references from `html-to-docx` to `@turbodocx/html-to-docx` in: the Project Structure note about `package.json`, the data-flow step that describes the DOCX export pipeline, the Export Workflow description, the Export Workflow Technologies line, and the Technology Used bullet.
