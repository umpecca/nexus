# T_112: Encrypt the QuickConnect bearer token at rest (safeStorage)

## Status

Done

## Summary

Stop persisting the QuickConnect bearer token in plaintext. Previously the token was saved inside
the per-OS-profile settings JSON in `localStorage` alongside the URL and path — a known exception
to the never-store-secrets posture, called out in `PRODUCT.md` §2.5 and listed as a future
iteration in §7. The token now moves out of `localStorage` entirely and is encrypted at rest by
the Electron main process using the OS-provided secure storage (`safeStorage`: DPAPI on Windows,
Keychain on macOS, libsecret on Linux). The non-secret URL and path remain ordinary local
settings. A legacy plaintext token is migrated into the encrypted store on first launch and then
removed from `localStorage`. When secure storage is unavailable, the token is used for the publish
but not persisted (never a plaintext fallback on disk).

## Implementation Notes

- Main process (`electron/main.cjs`):
  - Add `safeStorage` to the `require("electron")` destructure.
  - Add a small per-profile secret store backed by `quickconnect-tokens.json` under
    `app.getPath("userData")`: `getQuickConnectTokenStorePath`, `quickConnectProfileKey`,
    `readQuickConnectTokenStore` (tolerates a missing/corrupt file), `writeQuickConnectTokenStore`.
  - Register `quickconnect:get-token` (returns the decrypted token, or `""` when encryption is
    unavailable / no entry / decrypt fails) and `quickconnect:set-token` (empty token deletes the
    entry; non-empty stores `safeStorage.encryptString(token)` base64; never writes plaintext when
    encryption is unavailable). `set-token` returns `{ stored, encryptionAvailable }`.
- Preload (`electron/preload.cjs`) and types (`src/electron.d.ts`):
  - Expose `getQuickConnectToken(profileName)` and `setQuickConnectToken(profileName, token)`.
  - Add a `QuickConnectTokenSaveResult` type; the per-publish `QuickConnectInput.connection` still
    carries `token`.
- Settings (`src/lib/settings.ts`):
  - Drop `token` from `QuickConnectSettings`; `createDefaultQuickConnect` and `sanitizeQuickConnect`
    now handle only `url` and `path` (so any legacy token is stripped from re-saved settings).
  - Add `readLegacyQuickConnectToken(profileName)` to read a legacy plaintext token from
    `localStorage` once, for migration.
  - Update `src/lib/settings.test.ts`: `quickConnect` shape (`{ url, path }`) across the default,
    load, sanitize, and save assertions; drop-the-legacy-token-on-load case; new
    `readLegacyQuickConnectToken` cases.
- Renderer (`src/App.tsx`):
  - Hold the token in a separate in-memory `quickConnectToken` state.
  - In the profile-init effect, capture any legacy token synchronously before `setSettings` (which
    triggers the save effect that rewrites `localStorage` without it), then load the token via
    `getQuickConnectToken`, migrating the legacy token into the encrypted store when present.
  - Seed the dialog with `{ ...settings.quickConnect, token: quickConnectToken }`.
  - On submit, persist only `{ url, path }` in settings and store the token via
    `setQuickConnectToken`; keep sending the token in the publish `connection`.
- Docs: update `PRODUCT.md` (§2.5, §3.4, §3.4.x, §6, remove the §7 item), `ARCHITECTURE.md`
  (QuickConnect workflow, local-storage section, security notes, remove the implemented future
  iteration), and `docs/quickconnect.md` (bearer-token row + §7 security note).
